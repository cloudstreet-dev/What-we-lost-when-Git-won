# 1. What Version Control Is, and Why It's Plural

Version control, stripped to its operational core, is the practice of recording changes to a body of work in a way that lets you recover prior states, understand how the work evolved, and combine concurrent contributions without losing any of them. Every system covered in this book exists to provide some version of those three capabilities. They differ wildly in how they provide them, what they assume about who is making changes and from where, and what they make hard or easy.

That definition is intentionally bland. Bland is useful here, because most readers arrive carrying a much more specific picture. They picture commits, which are snapshots. They picture branches, which are pointers. They picture remotes, pushes, pulls, fetches, merges, rebases. They picture a directed acyclic graph of commits with letters or short hashes on it, possibly drawn by Git's `--graph` output. The picture is correct for Git. It is not the topic of the book. The topic is the wider design space inside which Git is one point.

To see the design space, start from the original problem. A body of work — text, code, images, datasets, schematics — is being changed by more than one person, possibly at the same time, possibly on more than one computer. The team needs to answer four kinds of question. *What did it look like before?* *Who changed it, when, and why?* *How do my changes interact with theirs?* *Which version is the one we are calling current?* Every question after that — branches, tags, releases, backports, reviews, audits, blame, bisect — is some specialization of one of those four.

Different fields have answered these questions differently for a long time. Manufacturing has document control systems with multi-stage approvals, immutable revisions, and signed change orders, because the document is a contract with regulators and customers and changing it without a paper trail is a felony in some jurisdictions. Wikipedia has a revision history that is both more and less than software version control: every edit is preserved with author and timestamp, but there is no notion of branch, no merge, no concept of work-in-progress that hasn't yet been published. Google Docs solves the concurrent-edit problem with operational transform on a centralized server and treats history as a side-effect of presence. Real-time collaboration tools like Figma use CRDTs over websockets to merge concurrent vector edits without human intervention. None of these are *wrong* answers to the version control question. They are answers shaped by their materials. Each material has a grain, and the version control system that fits the grain is the one that lives.

Software has properties that make some answers fit better than others. Source code is plain text that diffs cleanly along line boundaries — for the most part, in most languages, with caveats. It is produced by people who can themselves write tools, which means the version control system is also a software project being modified by people inclined to fork it. It is reorganized constantly: files renamed, code moved between modules, abstractions extracted and inlined, sometimes for reasons the next reader of history will need to understand. It is *reviewed*, before or after the change lands, by other humans whose attention is the scarce resource. And it accumulates: every codebase older than five years has files no one has read in years and tests no one has run in months and dependencies whose original authors have left the field. Version control for software has to handle that whole pile.

Within software alone, the design space is bigger than people who have only used Git tend to assume. The first axis is *centralization*. Some systems insist on a single canonical repository and treat checkouts as views into it; others let every working copy be a peer with equal rights, with synchronization a separate, optional concern. The second is *atomicity*. CVS committed file-by-file because it was glued together out of per-file RCS, and a CVS commit could half-fail in ways nobody could subsequently reconstruct; Subversion's first big idea was to make commits a single transaction. The third is *what a commit is*. Git records snapshots and computes diffs on demand. RCS recorded reverse deltas. Darcs records named patches with explicit commutation rules. Each choice has consequences for renames, merges, history rewriting, storage cost, and what kinds of question are easy or hard to ask of history later.

There is a fourth axis, harder to name. Call it *what the system thinks history is for*. Some systems treat history as an audit trail: it must be preserved, signed if possible, and never rewritten because the auditor will ask. Others treat history as a draft: shape it, edit it, fold the awkward fixups into the commit they should have been part of, and present a clean narrative for review. Mercurial defaulted to the audit-trail view; you had to opt into history rewriting through extensions and the system would warn you. Git defaulted to the draft view, and the practice of rebasing one's branch before opening a pull request became culturally near-universal. Neither stance is wrong. But each one shapes a team. A team that rewrites history will produce a different artifact than one that doesn't, for better and for worse, and the version control system the team uses is the original push that decided which culture they grew into.

That word *grew* matters. People talk about choosing a version control system as if it were a tool, like picking a particular text editor or build system. It is more like picking a programming language. The choice constrains what you can express easily, what idioms develop in the team, what kinds of error are common and what kinds are nearly impossible. Teams that have used Perforce for fifteen years on a game project have habits no Git team has, because the centralized exclusive-lock model produces a daily rhythm that Git does not. Teams that have used Mercurial professionally develop different reflexes than Git teams, because `hg pull` is not `git pull` and the difference shows up in everything from how you onboard a new engineer to how you run a release.

This is why the book is plural. There are not many version control systems because nobody has gotten it right yet. There are many because there are many real situations, with different artifacts, different team shapes, different regulatory and trust environments, different scales, different attention budgets. A solo developer working on a personal blog has different needs from a sixty-person game studio with terabytes of art assets and offline-capable laptops in three time zones. A nuclear regulator has different needs from a hackathon team. Some of those needs are shaped enough that the right answer is not Git, no matter how fluent everyone is in it.

It is also why the book uses the word *artifact* deliberately. Source code is the central case, and most of the time the artifact under version control will be a working tree of text files. But the moment you take seriously the question of what version control *is*, you have to reckon with the fact that people version game art, video edits, machine learning models, datasets, statistical analyses, legal contracts, build configurations, infrastructure-as-code, ML notebooks, schematics, BIM files, and CAD designs. A version control system that is genuinely good at code may be unusable for any of those, and a system that is excellent for one of them may be incomprehensible for code. Git is excellent for code and miserable for film. Perforce is excellent for film and overbuilt for a single-developer Rails app. Neither failure makes either system bad. They are tools shaped to materials, and the material always wins.

A note on the phrase *won the market*. Git won the market for hosted source code in the public, open-source corner of the software industry, and from there it spread back into closed-source corporate development under the gravitational pull of GitHub. It did not win the market for game development, nor for chip design, nor for film and visual effects, nor for several large enterprise codebases at companies whose names you would recognize. Inside Google and Meta, the version control story is more complicated than "we use Git": Google uses Piper for the monorepo with various Git-compatible front ends, Meta built and now maintains Sapling on top of a Mercurial-derived custom backend. Outside the public-OSS attention sphere, the variety of working version control systems in 2026 is greater than most working programmers realize. The book takes that as the starting point.

The rest of Part I sets up the analytical apparatus. Chapter 2 enumerates the design axes — centralization, atomicity, identity model, history-as-snapshot vs. history-as-patch, conflict resolution, locking, branching cost, history mutability, distribution model, and a half-dozen others — that the format chapters will use to compare systems. Chapter 3 specifies the exemplar scenario: a small fictional project, three contributors, a sequence of operations chosen to exercise every axis we care about. Every later chapter shows how the system in question handles those operations in its own idioms.

The format chapters do not translate. A Perforce chapter that explains `p4 sync` by saying "this is like `git pull` except—" is a chapter that has decided Git is the language and Perforce a dialect. The book refuses that move. Each system gets to be understood on its own terms first; comparison comes later, in the analytical chapters, where it can be done with care.

Two pieces of housekeeping before moving on. The first: the book uses *DVCS* for "distributed version control system" and *centralized VCS* for the older centralized model, even though the line between them is blurrier than the names suggest. Plastic SCM is centralized but with full-history clones that resemble a DVCS. Sapling is distributed in protocol but is most often deployed against a centralized server. The terms are useful labels, not strict categories. The second: every system mentioned has a current state of health, which the relevant chapter notes. Some are dead — GNU Arch is not coming back. Some are alive but quiet — Fossil is maintained, used, and largely ignored. Some are growing — Jujutsu is the youngest serious system in the book and is being adopted at a noticeable pace inside Google and outside it. The book tries to report the present accurately even when it makes the historical narrative less tidy.

The premise to take into Chapter 2 is this. Version control is plural because the design space is rich, the materials vary, and the social structures of the teams using these tools are not uniform. The marketplace winner at any moment is not the proof that other points in the design space are wrong; it is the proof that one combination of features and timing fit a particular set of conditions well enough to dominate. Conditions change. The conversation continues. A book about version control is not a guide to a settled topic; it is a partial map of an open one.
